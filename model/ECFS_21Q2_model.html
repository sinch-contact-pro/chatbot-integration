User properties

User's properties. Most of this data is generated by server, but user could change some of those with put request.

Server accepts user to update these resources: login_status, chat_address, language, username, first_name, last_name, alias, capabilities and presence. You can have all or only some of those included when you send put request

  • Response

HTTP status code 200

User's properties

Body

Media type: application/json

Type: object

Properties
  • id: required(string - pattern: ^[A-F0-9]{32}$)

    User's unique id (generated by Contact Center when you add new user)

    Example:

    EA88B8C7A1EA4E3BBDF25D1B4C3F628A
  • name: required(object)
    • first_name: required(string)

      First Name of the user as set in Contact Center System Configuration: Users and Role Management->Users->User X->Basics->First Name

      Example:

      John
    • last_name: required(string)

      Last Name of the user as set in Contact Center System Configuration: Users and Role Management->Users->User X->Basics->Last Name

      Example:

      Doe
  • username: required(string)

    Username of the user as set in Contact Center System Configuration: Users and Role Management->Users->User X->Authentication->Logon Name

    Example:

    john
  • alias: required(string)

    Alias of the user shown in conversations (ie: chat). By default this is received from Contact Center System Configuration: Users and Role Management->Users->User X->Basics->Chat Name. ECF client can update this for one user session.

    Example:

    Agent John
  • login_status: required(one of logged_in, logged_out, disconnected, anonymous)

    Login status of the user. Server updates this based on the user's login state

    Example:

    anonymous
  • work_status: required(one of ready, not_ready)

    The work status for the user. ready = user is ready to receive new interactions not_ready = user is not ready to receive new interactions

  • presence: required(string)

    Id of user's currently active presence profile. Refers to presence profiles

  • presence_profiles: required(array of object)

    All of this users available presence profiles

    Items: items

    • id: required(string - pattern: ^[A-F0-9]{32}$)

      Unique id for the presence profile

    • type: required(one of presence, absence, conference, unknown)

      Type of the profile. If presence type is absence, work_status change to ready is not accepted.

    • name: required(string)

      Presence profile name.

    • availability: (one of FREE, TENTATIVE, AWAY)

      Availability of the profile

    • settings: required(object)

      Presence profile details

      • reason: required(one of NONE, BUSY, LUNCH, MEETING, SICK, BUSINESSTRIP, TRAINING, VACATION, VISIT, PARENTAL, DAYOFF, DUTY, TEMPOUT, OFFDUTY, OFFICIAL, CHILDCARE, OTHER)

        Reason for the profile

      • default_behaviour: required(one of BUSY, FORWARD, USERFORWARD, CONFIVR, GSM, MOBILE, IVR, PASS, PBX, SWITCHBOARD, VOICEMAIL)

        What happens when the profile is enabled

      • duration: required(string - pattern: (NONE|REST_DAY|[0-9]+_(HOUR|DAY|MINUTE|WEEK)))

        Duration of the profile when enabled.

      • priority: required(string)

        Priority for the profile

  • chat_address: required(string)

    Email address of the user. By default this contains information what is inserted in System Configurator User and Role Management->Users->User X->Chat Address

  • email_address: required(string)

    If this is available, then user can send also outgoing emails.

  • extension: required(string)

    Phone extension number for the user (as configured in System Configurator) This number should be same as in User and Role Management->Users->User X->Basics->Number

  • language: required(one of EN, CS, DA, DE, ES, ET, FI, FR, HU, IT, JA, NL, NO, PL, PT, RU, SK, SV, TR, ZH)

    Language selected by agent server based on ContactCenter user configuration. User and Role Management->Users->User X->User Interface Language

  • permissions: required(object)

    User permissions as configured in Contact Center System Configurator. Configurations here can be set in: User and Role Management->Users->User X->User Rights->Type User Service->choose correct service

    • ecf_phone: required(boolean)

      Server sets this true only in case if user has MTD and PHONE rights and SIP - configurations are properly configured. If this is false, then server doesn't allow user to use phone (doesn't accept phoen calls, nor phone/sip_connected setting to true.

    • phone: required(boolean)

      This setting comes directly from system configurator. In case if user has phone - right, then this is true otherwise false.

    • mtd: required(boolean)

      This setting comes directly from system configurator. In case if user has mtd (= multi terminal desktop) - right then this is true, otherwise false.

    • video: required(boolean)

      This setting comes directly from system configurator. In case if user has video right, then this is true otherwise false.

    • email: required(boolean)

      This is true in case if user has ECF email available AND server can find at least one "from address" for user. From address can be either users own (set by SC) email address, or some of the queue addresses to which user has serve right. In case if server cannot find any email address for user, or ECF email is not supported for this user, this value is false. When this is false, server doesn't accept any email related events from client (= client cannot accept incoming email from UI etc.)

    • external_agent: required(boolean)

      This setting comes directly from system configurator. In case if user has external agent right (user service), then this is true otherwise false.

    • recording: required(one of enabled, disabled, hidden)

      This property will be generated by user's workspace. Reads following information from users workspace. Global Setting: /WorkspaceDocument/SystemInfo/ServerRecording User Setting: /WorkspaceDocument/UserInfo/ServerRecording





      Global settingUser SettingValue
      Not in useanyhidden
      User drivenNonehidden
      User drivenForced+hiddenhidden
      User drivenForceddisabled
      User drivenCtrl by userenabled
      User drivenCtrl by user, def onenabled
      Forcedanydisabled
      Forced + hiddenanyhidden
    • chat: required(boolean)

      This is true in case if user has ECF chat (Enable Multi-Chat configuration) available AND chat address is present for user. If this is false, server doesn't accept any chat related events from client, (= client cannot accept, pick incoming chat etc.), neither chats will be allocated for him by server

    • dpo: required(boolean)

      This setting comes directly from system configurator. In case if user has dpo (= data protection officer service) - right then this is true, otherwise false.

    • update_user: required(boolean)

      This setting comes directly from system configurator. In case if user has right to modify own User data then this is true, otherwise false.

    • update_user_settings: required(boolean)

      This setting comes directly from system configurator. In case if user has right to modify UserSettingsSet then this is true, otherwise false.

    • update_visible_a_number: required(boolean)

      This setting comes directly from system configurator. In case if user has right to modify UserSettingsSet and user setting item "CDT_CUI_SOL_PHONE_visibleanumber" is not denied (AllowOverride=1), then this is true, otherwise false.

  • terminal_id: required(string - pattern: ^[A-Z0-9]{32}$)

    Unique id for this client. Note that same user may open several client connections, every connection gets unique id. This information is usefull especially with phone calls, because every client will receive information about every interaction (even if audio 'stream goest to other client), with this information host application knows if this client is handling this interaction's voice).

  • phone: required(object)

    Phone configuration for user. This contains information about SIP websocket addresses, host name and extension number for this client connection.

    • sip_host: required(string)

      Host name which should be used when client registers to SIP server. Sip client should use this and extension when logging to server ([extension]@[sip_host])

    • sip_connected: required(boolean)

      Client should set this to true so that server knows when client has fully functional connection to sip server. In case if server sets this to false, client should concider that as forced disconnect (and should close SIP - connection). In case if user doesn't have ecf_phone permission, it's not allowed to update this resource.

    • extension: required(string - pattern: ^[a-zA-Z0-9]{32}$)

      Unique id for every client connection. Client should use this as "phone number" when it registers to SIP server. Client should pick host name from "sip_host". Client cannot try to make SIP - connection before it receives this, sip_host and sip_proxies - set.

    • sip_proxies: required(array of string)

      List of sip servers which could be used for sip to websocket connections. In case if this list is empty, client obviously cannot connect to SIP either.

    • external_agent: required(object)

      External Agent data

      • number: required(string)

        External agent phone number

      • permanent: required(boolean)

        Whether user is serving as external agent when not logged in

    • mtd: required(object)

      Multiterminal desktop (simultaneous ringing) data

      • active: required(boolean)

        Whether Multiterminal mode is active

      • in_gsm_active: required(boolean)

        Whether mobile phone number is used for inbound calls

      • in_gsm_number: required(string)

        Mobile phone number

      • in_hp_active: required(boolean)

        Whether hardphone number is used for inbound calls

      • in_hp_number: required(string)

        Hardphone number

      • in_other_active: required(boolean)

        Whether other phone number is used for inbound calls

      • in_other_number: required(string)

        Other phone number

      • in_virtual_active: required(boolean)

        Whether virtual phone number (user's phone extension) is used for inbound calls

      • out_terminal: required(one of GSM, HARDPHONE, OTHER, VIRTUAL)

        Terminal used for outbound calls

  • capabilities: required(object)

    Capabilities of the user. Client updates this based on what it knows about user's capabilities.

    • video_chat: (boolean)

      Video chat capability of the user. When host application knows that user is capable of using video, it should set this value to true. In case if client enables video capability, then server gives that information to Contact Center - part, and user can receive also video chats.

    • phone: (boolean)

      In case if host application notices that user has phone - capabilities (is able to use webrtc and accepts microphone etc), client can send this to server. In case if user has ecf_phone permission, server sends information to Contact Center - part that user can handle also phone calls after which server can provide calls to user.

  • supervising: required(object)

    Agent may deny (temporarily) supervising by modifying these values. By default supervising is accepted. Supervisor may change these only in case if usersettings has ECFUI_EnableSupervisorDenyButton == 1

    • deny_supervising: required(boolean)

      User sets this to true, if it doesn't want supervisor to start visoring at all.

    • deny_barge_in: required(boolean)

      User sets this to true, if it want's to disallow barge_in from supervisor.

    • deny_silent_listening: required(boolean)

      User sets this to true, if it want's to disallow silent_listening from supervisor.

    • deny_coaching: required(boolean)

      User sets this to true, if it want's to disallow coaching from supervisor.

    • deny_hangup: required(boolean)

      User sets this to true, if it want's to disallow hangup from supervisor. NOTE! This is not checked by server (= supervisor has right to access agent's model and thus supervisor can do any operations which can be done by agent --> UI may use this information to disable buttons)

    • deny_intercept: required(boolean)

      User sets this to true, if it want's to disallow intercept from supervisor. NOTE! This is not checked by server (= supervisor has right to access agent's model and thus supervisor can do any operations which can be done by agent --> UI may use this information to disable buttons)

    • deny_record: required(boolean)

      User sets this to true, if it want's to disallow record from supervisor. NOTE! This is not checked by server (= supervisor has right to access agent's model and thus supervisor can do any operations which can be done by agent --> UI may use this information to disable buttons)

get
post

In case if client wants to start new interaction, it should post that interaction to this resource.

Chat interaction post is allowed only from visitor side (= agent cannot start new chat currently)

Phone interaction will start new phone call in case if client has phone capability, and will pinpoint that phone call to this client.

Email interaction will create new outgoing email, and fill that interaction's draft - resource with given information. Email will not be sent before client set's email's draft status to "send". After sent, server removes email from model.

External interaction will create new "3rd party" interaction, and tell information about that to CEM. In case if CEM rejects interaction (fex. user already has ongoing interaction, or just received one), server sends error response back to this POST request.

Post interaction is also used for picking existing interaction. In that case it is mandatory to provide contact id and set status of interaction to "picked".

      post /interactions

      In case if client wants to start new interaction, it should post that interaction to this resource.

      Chat interaction post is allowed only from visitor side (= agent cannot start new chat currently)

      Phone interaction will start new phone call in case if client has phone capability, and will pinpoint that phone call to this client.

      Email interaction will create new outgoing email, and fill that interaction's draft - resource with given information. Email will not be sent before client set's email's draft status to "send". After sent, server removes email from model.

      External interaction will create new "3rd party" interaction, and tell information about that to CEM. In case if CEM rejects interaction (fex. user already has ongoing interaction, or just received one), server sends error response back to this POST request.

      Post interaction is also used for picking existing interaction. In that case it is mandatory to provide contact id and set status of interaction to "picked".

                Properties
                • status: (one of accepted, rejected, handled, wrapup, ended)

                  New status for interaction. In case if user wants to handle interaction, it should set interaction status to handled etc.

                • channel_status: (one of connected, muted, onhold, transferring, picking, outbound_accepted, outbound_rejected, outbound_skipped)

                  Channel status tells more detailed information about interaction's state. Other states are updated by server only, but these can be updated by user.

                  • connected: unhold / unmute call
                  • muted: mute call
                  • onhold: hold call
                  • transferring: transfer phone call to some other destination (destination should be included in this message)
                  • picking: tell to some other client that it should pick this call (OII uses this when it wants to command ECF client to pick call). This can be used by ECF client as well in case if it itself doesn't have phone - capable client, but has one open somewhere else, and wants to command that to pick the call.
                  • outbound_accepted: outbound preview ended, and can continue with real call
                  • outbound_rejected: outbound preview rejected
                  • outbound_skipped: outbound preview skipped
                • destination: (string)

                  User can give destination only in case if channel_status == transferring. Destination tells in that case to which destination user wants to transfer the interaction

                • attached_data: (object)

                  User can add additional call attached data in case of transfer to this interaction. Note that this data will not appear to interaction's model, and also this data should not overlap with current attached data.

                  • recording: (boolean)

                    User can start / stop recording by setting recording resource to true or false. In case if /users/me/properties/recording is hidden or disabled, setting of this value is not allowed. In case if interaction consent is false, client has possibility to override that consent. Client should open some separate query with which it asks user if (s)he really want to override consent or not, server doesn't check that anymore in this phase, if client sends recording=true for interaction which have consent=false, server will accept that, and passws AGENT_OVERRIDE - information to CEM when starting recording.

                    put /interactions/{externalid}

                    User can update external interaction's status or channel status. To accept incoming interaction user should set status to accepted. To reject incoming, status should be set to rejected. Difference between ended and handled is that when user set interaction ended, it means that interaction still stays in model (= is in "wrapup" state). When user sends handled, it means that also wrapup ended and interacction will be removed from model.

                    Since this is "external" interaction, putting status to accepted or rejected just updates model. In case if 3rd party CTI wants to actually disconnect call, it should set channel_status to disconnected. And in case if it want's to accept the call, it should set channel status to connected (in addition to status).
